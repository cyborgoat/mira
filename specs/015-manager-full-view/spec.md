# Feature Specification: 管理者自动获得个人+团队完整视图

**Feature Branch**: `main`

**Created**: 2026-05-26

**Status**: Draft

**Input**: 管理者登录后自动获得个人+团队完整视图（V03 + V04）

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理者问Mira 自动聚合全团队数据 (Priority: P1)

管理者在"问Mira"页面发起 AI 问答时，系统自动将管理者自己的数据与所有成员的数据合并作为 AI 上下文，无需任何切换操作。页面上方显示一行浅灰色小字「当前基于全团队数据回答」作为提示，Tab 键显示包含日报/周报及团队进展问题在内的四个快捷问题。

**Why this priority**: 问Mira 是管理者最高频的功能，自动聚合全团队上下文是管理者视图的核心价值主张。

**Independent Test**: 两个成员账号各有不同内容的笔记；以管理者账号登录，在问Mira 中提问"本周大家分别在做什么"，验证回答同时涵盖两个成员的信息，且页面上方显示「当前基于全团队数据回答」提示。

**Acceptance Scenarios**:

1. **Given** 管理者已登录，**When** 打开问Mira 页面，**Then** 输入框上方显示浅灰色小字「当前基于全团队数据回答」。
2. **Given** 管理者已登录，**When** 发送问题，**Then** AI 回答的上下文包含管理者自己的数据以及所有成员（非已注销）的数据。
3. **Given** 管理者已登录，**When** 在问Mira 输入框按 Tab 键，**Then** 显示四个预设问题：「写日报」「写周报」「本周各项目进展汇总」「哪个项目进展最慢」。
4. **Given** 成员已登录，**When** 打开问Mira 页面，**Then** 不显示「当前基于全团队数据回答」提示，Tab 键只显示个人预设问题（写日报/写周报/写月报）。

---

### User Story 2 - 管理者 Mira问（Lint）处理全团队数据 (Priority: P2)

管理者触发 Lint 扫描时，系统同时处理管理者自己的数据和所有成员的数据。生成的 LintSession 将问题分为「个人」（仅涉及管理者自身数据的矛盾）和「团队」（涉及多成员数据不一致的矛盾）两类。Mira问页面展示时，团队维度的问题卡片使用浅蓝色背景加以区分。

**Why this priority**: Lint 的跨成员矛盾检测是管理者视角下独有的高价值功能，帮助发现团队协作中的信息不一致。

**Independent Test**: 两个成员账号针对同一项目有不同进展描述；管理者触发 Lint，验证生成的 LintSession 中出现标注「团队」的跨成员矛盾问题，且「个人」类问题只来自管理者自身数据。

**Acceptance Scenarios**:

1. **Given** 管理者已登录，**When** 触发 Lint 扫描，**Then** 系统处理管理者自己及所有成员的数据，生成包含「个人」和「团队」两类问题的 LintSession。
2. **Given** 两个成员对同一项目有不一致描述，**When** 管理者查看 Lint 结果，**Then** 出现标注「团队」的问题卡片，描述两成员之间的矛盾。
3. **Given** Lint 结果页面展示，**When** 查看标注「团队」的问题卡片，**Then** 该卡片背景为浅蓝色，与「个人」类卡片视觉可区分。
4. **Given** 成员已登录，**When** 触发 Lint，**Then** 系统只处理该成员自己的数据，不生成跨成员问题。

---

### User Story 3 - 管理者统计页新增「团队概览」区域 (Priority: P2)

管理者在统计页面除了看到自己的个人统计外，页面底部还有一个默认展开的「团队概览」折叠区域，显示本周团队完成待办总数、活跃成员数，以及各项目的进度条。

**Why this priority**: 团队概览直接集成在统计页中，减少管理者在多个页面间切换的需要，提升信息密度。

**Independent Test**: 两个成员各自完成若干待办；管理者登录后打开统计页，验证「团队概览」区域显示正确的团队完成数和活跃成员数，各项目进度条与实际数据吻合。

**Acceptance Scenarios**:

1. **Given** 管理者已登录，**When** 打开统计页面，**Then** 页面底部显示「团队概览」折叠区域，默认展开。
2. **Given** 「团队概览」区域展开，**When** 查看数据，**Then** 显示「本周团队完成待办总数」「活跃成员数」和各项目进度条。
3. **Given** 成员已登录，**When** 打开统计页面，**Then** 不显示「团队概览」区域，页面只有个人统计内容。

---

### User Story 4 - 管理者知识图谱新增「团队知识图谱」区域 (Priority: P3)

管理者打开知识图谱页面时，除展示自己的个人 schema 外，页面底部新增「团队知识图谱」区域，聚合显示所有成员的项目、实体和决策信息。

**Why this priority**: 团队知识图谱为管理者提供跨成员的知识全景视图，是管理者视角下的高价值功能，但优先级低于 Lint 和统计。

**Independent Test**: 两个成员各有不同的项目和实体；管理者登录后打开知识图谱，验证「团队知识图谱」区域同时展示两个成员的项目和实体信息。

**Acceptance Scenarios**:

1. **Given** 管理者已登录，**When** 打开知识图谱页面，**Then** 页面底部显示「团队知识图谱」区域，聚合所有成员的 schema 数据。
2. **Given** 「团队知识图谱」区域，**When** 查看内容，**Then** 展示全成员的项目（Projects）、实体（Entities）、决策（Decisions）。
3. **Given** 成员已登录，**When** 打开知识图谱页面，**Then** 不显示「团队知识图谱」区域。

---

### User Story 5 - Rust 侧全面权限加固 (Priority: P1)

所有数据查询命令均包含统一的权限验证逻辑：未登录返回「请先登录」、越权访问他人数据返回「无权限访问其他账号的数据」、成员调用团队聚合命令返回「无权限访问团队数据」。前端统一处理未登录错误，自动跳转登录页。

**Why this priority**: 权限加固是安全基线，必须与所有功能同步完成，任何越权漏洞都会破坏整个多账号体系的可信性。

**Independent Test**: 以成员账号登录，通过开发工具直接发起 `get_team_stats` 命令调用，验证系统返回「无权限访问团队数据」错误而非返回数据。

**Acceptance Scenarios**:

1. **Given** 用户未登录，**When** 任意数据查询命令被调用，**Then** 系统返回「请先登录」错误，前端跳转登录页。
2. **Given** 成员 A 已登录，**When** 数据命令中传入成员 B 的 account_id，**Then** 系统返回「无权限访问其他账号的数据」错误。
3. **Given** 成员已登录，**When** 调用任意团队聚合命令（团队统计、团队 Lint、团队知识图谱聚合），**Then** 系统返回「无权限访问团队数据」错误。
4. **Given** 管理者已登录，**When** 调用团队聚合命令，**Then** 命令正常执行并返回数据。

---

### Edge Cases

- 所有成员均被软删除时，管理者的团队聚合命令应正常返回（管理者自身数据 + 空团队数据），不报错。
- 成员账号的笔记或任务为空时，该成员在团队聚合中贡献空数据，不影响其他成员数据的展示。
- 管理者触发 Lint 时，若所有成员数据均无矛盾，生成的 LintSession 中「团队」类问题列表为空，「个人」类问题列表展示管理者自身数据的 Lint 结果。
- 前端收到「请先登录」错误后，若用户正在编辑中（如正在输入笔记），应先提示「会话已过期」再跳转，避免数据丢失。

## Clarifications

### Session 2026-05-26

- Q: 管理者触发 Lint 后，生成的 LintSession 如何存储？ → A: 独立存储——新建一条 LintSession，与之前的个人 Session 并存；Mira问页面显示全部历史，用「个人」/「团队」标签区分 Session 类型。
- Q: 跨成员矛盾如何检测？ → A: AI 统一检测——全团队数据一次性交给 AI，由 AI 在同一次调用中识别并分类「个人」矛盾和「团队」跨成员矛盾。
- Q: 团队知识图谱中多成员同名项目/实体/决策如何展示？ → A: 去重合并——同名条目只显示一次，旁边附注参与成员名（如"项目X · 成员A、成员B"）。

## Requirements *(mandatory)*

### Functional Requirements

**V03 — 管理者完整视图**

- **FR-001**: 管理者登录后，问Mira 页面 MUST 在输入框上方显示一行浅灰色小字「当前基于全团队数据回答」。
- **FR-002**: 管理者登录后，问Mira 的 Tab 预设快捷问题 MUST 为四项：「写日报」「写周报」「本周各项目进展汇总」「哪个项目进展最慢」。
- **FR-003**: 管理者发起问Mira 问答时，AI 上下文 MUST 包含管理者自己的数据及所有未注销成员的 tasks 和 notes。
- **FR-004**: 管理者触发 Lint 时，系统 MUST 将全团队数据（管理者+所有未注销成员）一次性发送给 AI，由 AI 在单次调用中识别并分类矛盾；结果写入一条新建的独立 LintSession（类型标记为「团队」），原有个人 LintSession 不被覆盖或修改。
- **FR-004b**: Mira问页面 MUST 展示管理者全部 LintSession 历史（个人模式和团队模式），每条 Session 通过类型标签（「个人」/「团队」）加以区分。
- **FR-005**: Mira问页面 MUST 以浅蓝色背景渲染标注「团队」的问题卡片，与「个人」类卡片视觉区分。
- **FR-006**: 管理者统计页面 MUST 在个人统计区域下方显示「团队概览」折叠区域（默认展开），包含本周团队完成待办总数、活跃成员数、各项目进度条。
- **FR-007**: 管理者知识图谱页面 MUST 在个人 schema 下方显示「团队知识图谱」区域，聚合所有成员的项目、实体、决策信息；同名条目去重合并为一条，并附注参与成员名称（如"项目X · 成员A、成员B"）。
- **FR-008**: 成员登录后，问Mira 页面 MUST NOT 显示「当前基于全团队数据回答」提示，Tab 预设问题只显示个人版三项（写日报/写周报/写月报）。
- **FR-009**: 成员登录后，统计页面 MUST NOT 显示「团队概览」区域；知识图谱页面 MUST NOT 显示「团队知识图谱」区域。

**V04 — Rust 侧权限加固**

- **FR-010**: 所有数据查询命令 MUST 首先验证 session 是否存在；session 不存在时返回错误「请先登录」。
- **FR-011**: 单账号数据命令 MUST 验证请求的 account_id 与 session.account_id 一致；不一致时返回错误「无权限访问其他账号的数据」。
- **FR-012**: 团队聚合命令 MUST 验证 session.role 为"manager"；非管理者调用时返回错误「无权限访问团队数据」。
- **FR-013**: 前端所有数据命令的调用 MUST 处理「请先登录」错误，收到该错误时自动跳转到登录页。

### Key Entities

- **LintSession（含分类标签）**: 在现有 LintSession 结构的基础上，每条问题（LintQuestion）新增 `scope` 字段，取值「个人」或「团队」，用于前端渲染差异化样式。
- **团队聚合上下文**: 管理者问Mira 和 Lint 使用的数据集合，由管理者数据和所有非已注销成员数据组成，合并后截断至安全长度以满足 AI 上下文限制。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理者打开问Mira 页面，在 2 秒内看到「当前基于全团队数据回答」提示（零额外操作）。
- **SC-002**: 管理者发起问Mira 问答，AI 回答同时包含管理者和至少一个成员的数据信息（可通过两个账号各有独特关键词来验证）。
- **SC-003**: 管理者触发 Lint 后，生成的 LintSession 中「团队」类问题数量与实际跨成员矛盾数量一致（零遗漏、零误报）。
- **SC-004**: 成员调用任意团队聚合命令，100% 收到权限错误响应，无法获取其他账号数据。
- **SC-005**: 未登录状态下调用任意数据命令，100% 收到「请先登录」错误，前端自动跳转登录页，无数据泄露。
- **SC-006**: 管理者统计页「团队概览」数字与各成员实际任务完成数加总严格一致（误差为零）。

## Out of Scope

- 不引入新的账号管理功能（创建、删除、修改角色），这些由 specs/012 已实现。
- 不改变团队统计专属页面（specs/013 已实现的 TeamStatsPage），本 spec 只在统计页嵌入「团队概览」折叠区。
- 不实现实时推送或多客户端同步，所有数据在页面加载时一次性拉取。
- 不对 AI 问答的回答质量做保证，回答内容取决于 AI 模型能力。
- 不修改账号管理面板的 UI 或逻辑。

---

## Assumptions

- `LintQuestion` 结构目前无 `scope` 字段，需新增；现有 LintSession 数据不迁移（旧数据无 scope 字段时，展示时默认视为「个人」）。
- 团队聚合上下文截断阈值与 specs/013 中 `ask_team_mira` 使用的 8000 字符上限一致。
- 管理者问Mira 的「团队模式」与成员问Mira 的「个人模式」由同一个 `ask_mira` 命令根据 role 自动分支，无需两个独立命令。
- Mira问（Lint）的管理者团队模式由现有 `trigger_lint` 命令根据 role 自动分支，结果存入同一个 LintSession 结构，以 scope 字段区分问题类型。
- 知识图谱的「团队知识图谱」区域采用只读展示，不支持编辑或与个人图谱合并操作。
- 统计页「团队概览」的数据来源复用 `get_team_stats` 命令的返回值（specs/013 已实现），无需新增命令。
- 前端已有「请先登录」的跳转逻辑骨架（App.tsx 的 authState），本 spec 只需在 invoke 错误处理层统一接入。
