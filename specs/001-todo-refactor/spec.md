# Feature Specification: Todo 页面重构

**Feature Branch**: `001-todo-refactor`

**Created**: 2026-05-25

**Status**: Draft

**Input**: 重构「Todo」页面（原随手记），打造极简待办记录入口，用户 3 秒内完成记录。

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 快速新增待办 (Priority: P1)

用户打开应用，默认进入 Todo 页面。顶部有一个常驻输入框，用户直接输入内容、按回车，
待办即刻出现在列表顶部，输入框清空等待下一条输入。全程无需弹窗、无需额外点击。

**Why this priority**: 这是页面的核心价值——让记录比任何其他途径都快。此功能
独立可用，无需其他功能即可交付 MVP。

**Independent Test**: 打开应用后直接在输入框输入文字并按回车，确认待办出现在列表中，
输入框已清空，全程零弹窗。

**Acceptance Scenarios**:

1. **Given** 用户在 Todo 页面，**When** 在顶部输入框输入「买菜」并按回车，
   **Then** 「买菜」出现在列表顶部，输入框清空，焦点保持在输入框。
2. **Given** 输入框为空，**When** 用户按回车，**Then** 不创建任何待办，
   列表不变。
3. **Given** 输入框有内容，**When** 用户连续按两次回车，**Then** 只创建一条
   待办（防重复提交）。

---

### User Story 2 — 待办列表操作（完成 / 删除 / 行内编辑）(Priority: P2)

待办按创建时间降序排列。用户可将待办标记为「完成」（文字划线），可删除，
也可以点击待办文字直接在原位编辑（不弹窗），按回车或失焦保存。

**Why this priority**: 记录之后必须能管理。行内编辑比弹窗快，符合「极简」定位。

**Independent Test**: 在列表中分别执行完成、删除、行内编辑操作，确认每个操作
即时生效、无弹窗出现。

**Acceptance Scenarios**:

1. **Given** 列表中有一条待办「买菜」，**When** 用户点击「完成」按钮，
   **Then** 「买菜」文字出现删除线，该条目移入「已完成」折叠区。
2. **Given** 列表中有一条待办，**When** 用户点击「删除」按钮，
   **Then** 该条目从列表中移除，操作不可撤销（无二次确认弹窗）。
3. **Given** 列表中有一条待办「买菜」，**When** 用户点击文字，
   **Then** 该行切换为可编辑状态，用户修改为「买菜和水果」后按回车，
   显示更新后的内容。
4. **Given** 用户正在行内编辑，**When** 点击页面其他区域（失焦），
   **Then** 内容保存，退出编辑状态。

---

### User Story 3 — 已完成列表折叠展示 (Priority: P3)

已完成的待办独立折叠展示，默认收起。用户点击折叠头部展开，再次点击收起。

**Why this priority**: 保持主列表整洁是「极简」的关键，但历史记录也需要可访问。

**Independent Test**: 完成若干待办后，确认「已完成」区域默认收起，点击后展开
并显示所有已完成条目，再次点击收起。

**Acceptance Scenarios**:

1. **Given** 有 3 条已完成待办，**When** 页面加载，**Then**「已完成」折叠区
   显示在列表底部，默认收起，不显示具体条目。
2. **Given**「已完成」折叠区处于收起状态，**When** 用户点击折叠头部，
   **Then** 展开并显示所有已完成条目（带删除线）。
3. **Given** 没有任何已完成待办，**Then**「已完成」折叠区不显示。

---

### User Story 4 — Copilot 澄清卡片 (Priority: P4)

待办保存后，若 LLM Wiki 后端需要澄清，在待办列表顶部插入一张黄色背景的提问卡片。
用户可以文字回答后提交，或点「跳过」。回答或跳过后卡片消失，不影响待办列表操作。

**Why this priority**: 此功能增强 AI 知识库质量，但对主流程无依赖，可独立交付。

**Independent Test**: 模拟 Copilot 需要澄清的情况，确认卡片出现在列表顶部（非遮挡），
回答或跳过后卡片消失，待办列表仍可正常操作。

**Acceptance Scenarios**:

1. **Given** 用户新建了一条待办并触发了 Copilot 澄清，**When** 待办保存完成，
   **Then** 列表顶部出现黄色背景卡片，标题「Mira 想确认一件事」，
   下方有提问内容、文字输入框和「跳过」按钮。
2. **Given** Copilot 卡片可见，**When** 用户输入回答并提交，
   **Then** 卡片消失，回答被发送给后端，待办列表保持完整。
3. **Given** Copilot 卡片可见，**When** 用户点击「跳过」，
   **Then** 卡片消失，不发送任何内容给后端。
4. **Given** Copilot 卡片可见，**When** 用户继续在输入框新建待办，
   **Then** 新建操作正常完成，卡片不阻断操作。

---

### Edge Cases

- 输入框输入极长文字（超过 500 字符）时，截断并提示或限制输入。
- 多条 Copilot 澄清同时触发时，每次只展示一张卡片，前一张消失后再出现下一张。
- 已完成列表为空时，折叠区完全不渲染（不占位）。
- 应用离线时，新建待办应有合理的本地暂存或错误提示（中文）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 页面 MUST 是应用启动后的默认（首选）标签页。
- **FR-002**: 顶部输入框 MUST 常驻显示，placeholder 文案为「记录一件事…」。
- **FR-003**: 用户输入非空内容后按回车，系统 MUST 创建待办并将其插入列表顶部，
  输入框随即清空，焦点留在输入框。
- **FR-004**: 待办列表 MUST 按创建时间降序排列。
- **FR-005**: 每条待办 MUST 提供「完成」按钮；点击后文字出现删除线，条目移入
  已完成区。
- **FR-006**: 每条待办 MUST 提供「删除」按钮；点击后立即移除条目，无二次确认弹窗。
- **FR-007**: 点击待办文字，MUST 进入行内编辑模式；按回车或失焦 MUST 保存更改。
- **FR-008**: 已完成待办 MUST 折叠展示，默认收起；折叠区为空时不渲染。
- **FR-009**: 待办列表为空时，MUST 显示空态文案「今天还没有待办，记录一件事吧」。
- **FR-010**: 系统 MUST 在待办保存后向后端查询是否有 Copilot 澄清问题。
  若有，MUST 在列表顶部展示黄色背景卡片，标题「Mira 想确认一件事」。
- **FR-011**: Copilot 卡片 MUST 提供文字输入框（用于回答）和「跳过」按钮；
  回答提交或跳过后卡片 MUST 消失。
- **FR-012**: Copilot 卡片 MUST NOT 以模态弹窗形式出现，MUST NOT 遮挡待办列表。
- **FR-013**: 同一时刻最多展示一张 Copilot 卡片；多个澄清请求队列处理。
- **FR-014**: 页面 MUST NOT 展示优先级标签、截止日期或分项目过滤控件。

### Key Entities

- **Todo（待办）**: 用户创建的一条记录，包含内容文字、创建时间、完成状态、
  优先级（字段保留，页面不展示）、截止日期（字段保留，页面不展示）。
- **CopilotQuestion（澄清问题）**: 由后端（LLM Wiki）生成，包含问题文本；
  与触发它的 Todo 关联。用户的回答发送回后端，跳过则丢弃。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从输入框输入内容到待办出现在列表，操作路径不超过 1 次用户动作
  （按回车），耗时感知不超过 3 秒。
- **SC-002**: 完成、删除、行内编辑三项操作均无弹窗，操作路径不超过 2 次点击。
- **SC-003**: Copilot 卡片出现时，用户无需关闭卡片即可继续在输入框新建待办，
  卡片与主功能的操作完全不阻断。
- **SC-004**: 页面可见元素数量在无待办时 ≤ 2 个区域（输入区 + 空态提示）；
  有待办时不引入任何「优先级 / 截止日期 / 过滤器」相关视觉元素。

## Assumptions

- 待办数据通过 Tauri Rust 命令与本地 SQLite 存储，前端仅调用 invoke，不直接操作数据库。
- Copilot 澄清问题由后端主动推送或前端在保存后轮询，具体机制由后端规范决定；
  前端只需处理「有问题」和「无问题」两种状态。
- 本次重构不改变 Todo 的底层数据结构，只调整前端展示和交互。
- 优先级和截止日期字段在数据层保留，但本页面的任何 UI 渲染均不涉及这两个字段。
- 应用已有导航框架（五标签页），本次只替换 Todo 标签页的内容组件。
- 「应用离线」场景的具体处理策略由现有错误处理基础设施决定，本 spec 不重新定义。
