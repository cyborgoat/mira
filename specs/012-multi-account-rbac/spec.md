# Feature Specification: 本地多账号系统与账号管理页面

**Feature Branch**: `012-multi-account-rbac`

**Created**: 2026-05-26

**Status**: Draft

**Input**: 为 Mira 添加本地多账号系统和账号管理页面，支持多账号数据隔离、角色权限控制（成员/管理者），
以及管理者专属的账号管理界面。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 成员账号登录与数据隔离 (Priority: P1)

张三（成员）打开 Mira，看到账号选择列表，点击自己的头像卡片后输入密码，
成功登录后进入自己专属的工作空间。张三看到的待办、笔记和其他数据与李四的完全分离，
互不干扰。

**Why this priority**: 这是多账号系统的核心价值，若数据不隔离，其他功能毫无意义。

**Independent Test**: 创建两个成员账号，分别写入不同的待办事项，切换登录后验证各自只能看到自己的数据。

**Acceptance Scenarios**:

1. **Given** 系统已有多个账号，**When** 用户打开应用，**Then** 登录页显示所有账号的卡片列表，每张卡片包含头像占位图、姓名和角色标签。
2. **Given** 用户点击自己的账号卡片，**When** 输入正确密码并点击「登录」，**Then** 跳转至主界面，导航栏按角色显示正确页面（成员见 6 页，管理者见 7 页）。
3. **Given** 用户输入错误密码，**When** 点击「登录」，**Then** 显示中文错误提示「密码错误，请重试」，停留在密码输入界面。
4. **Given** 登录成功，**When** 用户查看任意数据页面（待办/笔记/统计），**Then** 只显示该账号自己的数据，不包含其他账号的任何记录。

---

### User Story 2 - 管理者创建和管理账号 (Priority: P2)

管理者王五登录后，进入「账号管理」页面，可以新建团队成员账号，为他们设置姓名、密码和角色，
也可以修改现有账号的角色，或在确认后删除不再需要的账号。

**Why this priority**: 账号管理是多账号系统的核心管控入口，成员账号的存在依赖管理者创建。

**Independent Test**: 管理者创建一个新成员账号，验证新账号出现在登录页账号列表且能成功登录。

**Acceptance Scenarios**:

1. **Given** 管理者已登录，**When** 查看导航栏，**Then** 导航中显示「账号管理」作为第 7 项，成员登录时该项不可见。
2. **Given** 管理者进入账号管理页，**When** 点击「新建账号」，**Then** 显示表单：姓名（必填）、密码（必填）、角色选择（团队成员/团队管理者）。
3. **Given** 填写完整表单并提交，**When** 确认，**Then** 新账号出现在账号列表，并立即出现在登录页的账号卡片中。
4. **Given** 管理者在账号列表查看某账号，**When** 点击「修改角色」，**Then** 显示角色选择器，保存后该账号角色立即更新。
5. **Given** 管理者点击某账号的「删除账号」（非自己），**When** 确认弹窗显示「删除后该账号所有数据将被清除」，**Then** 点确认后账号及其全部数据被删除，该账号从登录列表消失。
6. **Given** 管理者尝试删除自己的账号，**When** 点击删除按钮，**Then** 显示「无法删除当前登录账号」，操作被阻止。

---

### User Story 3 - 现有数据迁移至默认账号 (Priority: P3)

首次引入多账号功能时，系统检测到没有 accounts.json，自动将现有业务数据归属至
一个默认管理者账号（「默认账号」），用户无感知地进入多账号模式，原有数据完整保留。

**Why this priority**: 保护现有用户的数据，避免升级丢失内容；但只影响首次升级场景。

**Independent Test**: 在已有业务数据（待办/笔记）的环境中运行新版应用，验证登录页出现「默认账号」卡片，
登录后数据完整无缺失。

**Acceptance Scenarios**:

1. **Given** 旧版数据存在但 accounts.json 不存在，**When** 启动应用，**Then** 系统自动创建「默认账号」（角色：管理者），并将现有所有业务数据的 account_id 设为该账号的 UUID。
2. **Given** 迁移完成，**When** 用户在登录页看到「默认账号」卡片，**Then** 默认账号无密码要求（或使用空密码直接登录），提示用户在账号管理中设置密码。
3. **Given** 迁移后，**When** 其他新账号创建，**Then** 默认账号数据不受影响，数据隔离正常工作。

---

### Edge Cases

- 账号列表为空时（全新安装无 accounts.json），应显示「暂无账号，请联系管理者创建账号」，并提供「以默认管理者身份初始化」的引导。
- 删除的账号若持有活跃 AI 任务，任务应随账号数据一并清除。
- 密码为空时不允许创建账号（迁移的默认账号除外）。
- 当设备上只剩一个管理者账号时，不允许将其角色降级为成员，防止锁死。
- 账号名称重复时，创建应成功（以 UUID 区分），但在 UI 上显示警告提示。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 登录页 MUST 显示所有本地账号的卡片列表，每张卡片包含头像占位、姓名和角色标签。
- **FR-002**: 用户 MUST 能通过点击账号卡片 → 输入密码 → 点击「登录」完成认证。
- **FR-003**: 登录成功后，Session MUST 携带 `account_id`、`name`、`role` 和 `token`。
- **FR-004**: 所有数据读写操作 MUST 以当前 Session 的 `account_id` 作为过滤条件，成员不得访问其他账号数据。
- **FR-005**: 「账号管理」页面 MUST 仅在管理者 Session 下出现于导航栏（第 7 项），成员 Session 中该页面的路由和组件 MUST NOT 被渲染。
- **FR-006**: 管理者 MUST 能创建新账号，必填字段：姓名、密码、角色（团队成员/团队管理者）。
- **FR-007**: 管理者 MUST 能修改任意账号的角色（自己账号除角色降级限制外）。
- **FR-008**: 管理者 MUST 能删除任意账号（自己除外），删除前 MUST 显示二次确认弹窗。
- **FR-009**: 删除账号操作 MUST 级联删除该账号的所有业务数据（tasks、notes、wiki_schema、lint_sessions）。
- **FR-010**: 系统 MUST 在检测到 accounts.json 不存在时自动执行数据迁移，将现有数据归属至默认管理者账号。
- **FR-011**: 账号信息 MUST 存储在独立的 accounts.json，不与业务数据混合。
- **FR-012**: 所有账号密码 MUST 以 bcrypt 哈希存储，明文密码 MUST NOT 出现在任何存储层或日志中。
- **FR-013**: 当设备上只剩一个管理者时，系统 MUST 阻止将该管理者降级为成员，并显示中文错误提示。
- **FR-014**: 所有 UI 文案 MUST 为简体中文，包含所有提示、错误信息、按钮和标签。

### Key Entities

- **Account**: 账号实体，包含 `id`（UUID）、`name`（显示名）、`password_hash`（bcrypt）、`role`（"member"/"manager"）、`created_at`（ISO 8601）。
- **Session**: 认证会话，包含 `account_id`、`name`、`role`、`token`（随机 token，仅存于内存）。
- **AccountInfo**: 对外暴露的账号信息，包含除 `password_hash` 外的所有字段。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 拥有多个本地账号的用户能在 30 秒内完成账号选择和登录操作，全程无需网络连接。
- **SC-002**: 管理者能在 2 分钟内完成新账号创建，新账号立即出现在登录页。
- **SC-003**: 两个不同账号的数据 100% 隔离 — 任何跨账号数据泄露视为严重错误。
- **SC-004**: 现有用户升级至多账号版本后，原有数据 100% 保留，无数据丢失。
- **SC-005**: 删除账号后，该账号的所有业务数据在 5 秒内完全清除，存储空间释放。
- **SC-006**: 管理者账号的所有管控操作（创建、修改、删除）仅通过「账号管理」页完成，成员账号无法触达这些操作。

## Assumptions

- 这是本地演示版，不涉及网络、云端同步或跨设备账号共享。
- accounts.json 存储在与业务数据相同的工作目录下，与 mira-api.sqlite3 同级。
- 现有数据迁移时，默认账号使用空密码（允许直接登录），管理者可在账号管理中为其设置密码。
- 头像仅为占位符（首字母头像），不支持自定义头像上传。
- 账号 token 为随机字符串，仅用于同一 session 内的身份验证，关闭应用即失效。
- 管理者的跨账号聚合查询能力在本期规划中不包括前端可视化（如「所有人的待办总数」），仅确保数据架构支持。
- 账号名允许重复（以 UUID 区分），但 UI 会在重名时显示警告。
